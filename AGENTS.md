## One-paragraph summary

`semgrep-rules-no-animal-violence` is a Semgrep rule library that detects speciesist language — animal-harming idioms, speciesist metaphors, and dehumanizing tech jargon — in source code, documentation, comments, and configuration files. It contains four rule files (generic, Python, JavaScript/TypeScript, Go) covering 62+ patterns, all auto-generated by `project-compassionate-code` from the canonical `no-animal-violence` dictionary. The rules run in CI via GitHub Actions, as pre-commit hooks, and are loaded at runtime by live MCP infrastructure (`mcp-server-nav-language`, `lbr8-mcp-constraints`, `mcp-server-aha-evaluation`). Rule IDs are stable identifiers used by the audit-to-dispatch system to key fix tasks.

## Architecture

### Two-layer rule design

The ruleset uses two complementary approaches:

**Layer 1 — Generic rules** (`languages: [generic]`)
Semgrep treats each file as plain text and applies regex patterns (`pattern-regex`). This catches speciesist language anywhere it appears: inside comments, documentation, config files, Markdown, shell scripts. It cannot do autofix because there is no AST to anchor replacements. Produces the highest recall.

**Layer 2 — Language-specific rules** (Python, JS/TS, Go)
Semgrep parses the AST and uses `metavariable-regex` to match inside string literal nodes only. This eliminates false positives from import paths, function names, and system calls. It supports `fix-regex` autofix. Cannot match inside comments because the AST excludes them.

For full coverage, always run both layers.

### Semgrep rule format

Each rule is a YAML object in the `rules:` list:

```yaml
rules:
  - id: <unique-rule-id>
    languages: [generic | python | javascript | go]
    severity: ERROR | WARNING | INFO
    message: <human-readable explanation and suggested alternative>
    patterns:
      - pattern-regex: <regex>          # generic mode
      # or
      - pattern: $STR                   # AST mode
        metavariable-regex:
          metavariable: $STR
          regex: <regex>
    fix-regex:                          # AST mode only
      regex: <pattern to replace>
      replacement: <replacement text>
    metadata:
      category: inclusive-language
      subcategory: animal-violence
      alternative: <preferred replacement phrase>
      references:
        - <doi or url>
```

### Rule ID naming

```
animal-violence.<idiom-slug>                          # generic
animal-violence.<language>.string.<idiom-slug>        # language-specific
```

Rule IDs are stable identifiers used by live MCP infrastructure. Do not change them without coordinating with `mcp-server-nav-language` and verifying the audit-to-dispatch key mappings (decision #37, 2026-04-11).

### Auto-generation pipeline

```
no-animal-violence (canonical dictionary)
    └── project-compassionate-code (generator)
            └── semgrep-rules-no-animal-violence (this repo)
                    └── downstream consumers
                            ├── mcp-server-nav-language (loads YAML at startup)
                            ├── lbr8-mcp-constraints (bundles subset offline)
                            ├── mcp-server-aha-evaluation (Stage 1 content filter)
                            ├── desloppify (scores repos)
                            └── open-paws-platform CI
```

## Integration points

### Upstream

- **no-animal-violence** — canonical pattern dictionary. All rule additions start here.
- **project-compassionate-code** — generates the four rule YAML files. Do not edit YAML directly.

### Downstream (live as of 2026-04-11)

- **mcp-server-nav-language** — loads `rules/animal-violence-generic.yaml` (and language-specific YAMLs) at startup. Exposes `check_language`, `check_file`, `list_rules` tools. Sub-10ms response. Integrated into Gary MCP hub Phase 3. Sensitive to rule ID changes and YAML structure changes.
- **lbr8-mcp-constraints** — bundles 12 offline NAV patterns sourced from the generic rules file. Wraps MCP tool handlers with constraint middleware.
- **mcp-server-aha-evaluation** — uses NAV rules as Stage 1 of a two-stage content evaluation pipeline.
- **audit-to-dispatch** (decision #37) — NAV violations found during ecosystem audits auto-dispatch as fix tasks, keyed by rule IDs from this repo.
- **desloppify** — embeds speciesist language detection as a scoring dimension.
- **Open Paws platform CI** — integration planned, not yet shipped (see `ecosystem/integration-todos.md` §27a).

## Safe vs. risky changes

### Safe to change

- `tests/test_placeholder.py` — structural test improvements are safe and encouraged.
- `CONTRIBUTING.md`, `README.md`, `AGENTS.md` — documentation updates, no functional impact.
- `.github/workflows/ci.yml` — CI pipeline changes. Test in a branch first.
- `.pre-commit-config.yaml` — pre-commit hook config. No impact on rule content.
- `.yamllint.yml`, `ruff.toml` — linter configuration.

### Requires care

- **Adding new rules** — Safe if added to all four files (generic + 3 language-specific) consistently. Verify with `semgrep --config rules/ --validate` and `pytest tests/`.
- **Modifying existing rule messages** — Safe; messages are not used as keys by any downstream consumer.
- **Modifying `fix-regex` replacements** — Test against real code before merging. Incorrect regex can corrupt string literals.

### Risky — coordinate before changing

- **Changing rule IDs** — Breaks `mcp-server-nav-language` (requires server restart after change), and invalidates audit-to-dispatch task keys. Announce changes in the MCP infra channel first.
- **Changing YAML structure or field names** — `mcp-server-nav-language` parses this YAML directly. Schema changes require a coordinated deploy.
- **Removing rules** — Downstream consumers may rely on specific rule IDs being present. Check `mcp-server-nav-language` and `audit-to-dispatch` mappings before removing.

### Do not do

- Edit rule YAML files directly with the intent of keeping the change. They will be overwritten by `project-compassionate-code` on the next generator run. Changes to patterns must go through the canonical dictionary in `no-animal-violence`.
- Skip `semgrep --config rules/ --validate` before committing. Invalid YAML breaks all downstream consumers.
- Add language-specific rules without a corresponding generic rule. The generic rule is required for comment and doc coverage.
